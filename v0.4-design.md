# franky-do v0.4 design — interactive conversation

Three feature areas planned for the v0.4 cut, all aimed at making the
bot's behavior in a Slack thread feel like a real interactive
collaborator instead of a black-box request/response endpoint.

## A — Emoji status indicators

### A.1 Goal

When the user `@`-mentions the bot, surface what's happening in real
time using **reactions on the user's own message**. Today the only
visible signal is the placeholder reply (`_thinking…_`) and the
streamed text. Reactions are zero-friction, language-neutral, and
work in both DMs and channel mentions.

### A.2 Lifecycle (proposed)

| Phase | Reaction added | Reaction removed |
|---|---|---|
| Mention received, before placeholder posted | `👀` (`:eyes:`) | — |
| Agent worker started | `💭` (`:thought_balloon:`) | — |
| Tool execution starts | `🔧` (`:wrench:`) | `💭` |
| Tool execution ends | (revert to `💭` if more turns) / `✅` if done | `🔧` |
| Turn end with text reply, no more turns | `✅` (`:white_check_mark:`) | `💭` |
| `agent_error` | `❌` (`:x:`) | all transient |

**Why react on the user's mention, not the bot's reply.** Slack's
mobile clients animate reactions on the source message; users see the
state without scrolling. Reactions on the bot's `_thinking…_` reply
would be subtler. Also matches Phase 7's reaction-as-control where
reactions on EITHER the user mention or the bot reply resolve back to
the thread (we already have that machinery via `reply_anchors`).

### A.3 Open questions

#### A.3.1 — Per-tool emojis vs single 🔧 for all

**Proposal:** single `🔧` for any tool. Per-tool emojis (read=📖,
find=🔍, write=✏️, edit=📝, ls=📂, grep=🔎) are nicer but flicker
under parallel-tool execution and don't add much over the wrench.

**Decision:** single `🔧` for all tools.

#### A.3.2 — Slack scope: `reactions:write` is required

The current manifest only has `reactions:read`. Adding `reactions:write`
forces an app reinstall in every workspace. **Proposal:** add the scope;
include a migration note in `TESTING.md` so operators know to reinstall
when upgrading from v0.3 → v0.4.

**Decision:** add the scope in the app manifest as well; include a migration note in `TESTING.md` so operators know to reinstall when upgrading from v0.3 → v0.4.

#### A.3.3 — Rate-limit handling

Slack's `reactions.add` is rate-limited at "Tier 3" (~50/min/workspace).
A long agent run with many tool calls could plausibly hit this on a
bot serving multiple concurrent threads. **Proposal:** if `reactions.add`
returns 429, log at warn and skip — the user-visible degradation is
"reactions stop appearing" (not "the bot fails"). Don't retry.

**Decision:** log at warn and skip if `reactions.add` returns 429. Also consider to not react immediately to any tool call or response keep a buffer of pending reactions and flush them at the end of the turn, this way we can reduce the number of API calls and avoid hitting the rate limit in the first place.

#### A.3.4 — Cleanup on completion

After `✅` lands, do we leave the trail of historical reactions
(`👀` + `✅`) or clean up? **Proposal:** leave them. Removing reactions
adds another API call per state change; the trail also serves as a
visible history of "yes, the bot saw this and finished it."

**Decision (revised, v0.3.4):** **keep only the latest state
emoji.** Each state transition removes the previous state's
reaction before (or after) adding the new one. So the user's
mention message at any moment shows exactly one of `👀` /
`💭` / `✅` / `❌` — the bot's *current* status, not a history.

Why the change:
- The original "leave the trail" decision optimized for one API
  call per state. In practice the trail is more confusing than
  helpful: a returning user sees `👀 💭 ✅` and has to read
  three emojis instead of one to figure out "is it done?".
- The mobile-client animation argument (A.2 rationale) only
  works when there's a *single* reaction transitioning — three
  static reactions is just clutter.
- The transcript itself + the placeholder reply already provide
  the audit trail. The reactions are a *live status* indicator,
  not a log.
- API cost: one extra `reactions.remove` per transition. Three
  transitions per mention (👀→💭, 💭→✅, optional →❌). Still
  well under Slack's Tier 3 budget.
- Failure mode: if `reactions.remove` 429s or otherwise fails,
  log warn and skip. Worst case the user sees two stale emojis
  for one mention — same UX as the original "leave the trail"
  decision, so the floor is no worse than before.

Implementation note: order matters. Add the new reaction
*before* removing the old one (in case the remove fails — at
least the user sees the new state). On the rare race where add
succeeds and remove fails, we log and accept the duplicate
emoji.
### A.4 Wire shape

- **Web API additions** (`src/slack/web_api.zig`): `reactionsAdd(.{
  channel, timestamp, name })` and `reactionsRemove(.{ channel,
  timestamp, name })`. Wrap `reactions.add` / `reactions.remove`
  endpoints; both return `{ ok: bool, error: ?string }`.
- **Subscriber** (`src/stream_subscriber.zig` or new `src/reactions_subscriber.zig`):
  a second `franky.agent.Agent.subscribe` callback that listens for
  `tool_execution_start` / `tool_execution_end` / `turn_start` /
  `turn_end` / `agent_error` and invokes the Web API helpers
  asynchronously (don't block the agent's broadcast loop on the
  network call).
- **Manifest** (`slack-app-manifest.yaml`): add `reactions:write`.

### A.5 Tests

Reuse the loopback Slack server pattern from
`stream_subscriber.zig`'s existing tests. Three test cases:

1. **Happy path** — fire the relevant `AgentEvent`s, assert
   `reactions.add` requests captured in source-event order.
2. **Rate-limit handling** — server returns 429; assert no panic and
   subsequent calls still attempt.
3. **Cancellation mid-run** — `agent_error` lands; assert `❌` is
   added and pending `🔧` removal still completes.

### A.6 Effort

~150-250 LOC including the new subscriber + tests. Roughly half a
day of work.

## B — Slack-side tool permission prompting

### B.1 Goal

Today the bot ships a fixed tool set (`read/write/edit/ls/find/grep`)
with `bash` deliberately omitted (§6.4). The v1.11.x permission
overlay (`--prompts` + `tool_permission_request` event) is **not
wired** in franky-do (§16.0). For v0.4, expose it through Slack so
operators who want belt-and-braces gating on `write`/`edit` (or who
want to selectively re-enable `bash`) can flip a flag and have the
bot ask in-thread before each call.

This mirrors what franky's web UI does in `--mode proxy` — a yellow
modal pops up with allow / deny / always-allow / always-deny
buttons. Slack equivalent: a thread-reply with reaction-driven
resolution.

### B.2 Surface (proposed)

When the agent's `before_tool_call` hook fires `ask`, the bot:

1. Posts a thread-reply formatted as:

   ```
   ⚠️ The agent wants to call:
       write(path="src/foo.zig", content=...)

   React to this message:
       ✅  allow once       ⏩  always allow
       ❌  deny once        🛑  always deny
   ```

2. Adds the four reactions on its own reply so users have one-tap
   resolution.
3. Suspends the agent worker via the existing
   `permissions.PermissionPrompter.requestAndWait` (§5.11.1) until a
   `reaction_added` event resolves the prompt.
4. On reaction: route through `prompter.resolve(call_id, resolution)`
   and let the agent loop continue; remove the four reactions from
   the prompt message; post a status line ("✓ allowed by `<@user>`"
   or "✗ denied by `<@user>`").

### B.3 Open questions

#### B.3.1 — Opt-in mechanism

**Proposal:** `--prompts` CLI flag + `FRANKY_DO_PROMPTS=1` env. Off by
default; Slack-bot users who care can flip it. Exact mirror of franky's
`--prompts`. Matches the v1.19.0 settings-layer overlay (a project's
`settings.json` can also flip `prompts: true`).

**Decision:** Should be an Opt-Out mechanism, enabled by default. The permission prompting is a critical security feature, and having it enabled by default ensures that all users benefit from it without needing to take additional steps. Operators who want to disable it can do so with the `--no-prompts` flag or `FRANKY_DO_PROMPTS=0` env var.

#### B.3.2 — Default policy when `--prompts` is on

**Proposal:** match franky's `permissions.defaultPolicy`:
`read`/`ls`/`find`/`grep` auto-allow (read-family is safe); `write` /
`edit` / `bash` (if re-enabled) ask. CLI overlay (`--allow-tools` /
`--deny-tools`) lets operators tighten or loosen per-tool.

**Decision:** match franky's `permissions.defaultPolicy`.

#### B.3.3 — Prompt timeout

A user might `@`-mention the bot from mobile, then forget. The agent
worker would suspend forever waiting for a reaction. **Proposal:** 5
minute prompt timeout (configurable via
`FRANKY_DO_PROMPT_TIMEOUT_MS`). On timeout, resolve as `deny_once`
with a posted explanation ("⏱ no response in 5 minutes — denied; ask
again to retry").

**Decision:** 10 minute prompt timeout (configurable via `FRANKY_DO_PROMPT_TIMEOUT_MS`). On timeout, resolve as `deny_once` with a posted explanation ("⏱ no response in 5 minutes — denied; ask again to retry").

#### B.3.4 — Who can resolve

**Proposal:** **anyone in the thread**. Multi-user threads are
expected; restricting to "only the original `@`-mentioner" introduces
politics ("alice asked, bob can't approve"). Audit the resolver in
the status line so it's transparent.

**Alternative**: only thread participants (anyone who's posted in the
thread before the prompt). Rejected as too restrictive — bots in
incident channels often have new joiners reacting.

**Decision:** Only the original `@`-mentioner can resolve. This keeps the permission control tight and avoids potential conflicts in multi-user threads.

#### B.3.5 — Persistent always-allow / always-deny scope

franky's `--remember-permissions` writes always-resolutions to
`$FRANKY_HOME/permissions.json`. **Proposal for franky-do:** same env
var (`FRANKY_DO_REMEMBER_PERMISSIONS=1`), file at
`$FRANKY_DO_HOME/permissions.json`. Scope is *workspace-wide* (one
file per franky-do instance, not per Slack workspace). Per-team or
per-thread scope is a v0.5 follow-up.

**Decision:** Enable persistent always-allow / always-deny by default. Operators can disable it with `FRANKY_DO_REMEMBER_PERMISSIONS=0`.
Keep the scope workspace-wide for v0.4 to keep the implementation simpler. Per-team or per-thread scope can be considered for v0.5 if there's demand.

#### B.3.6 — Bash re-enablement

Today `bash` is *not registered* (`main.zig:395`'s `tools_arr` excludes
it; §6.4 explains why). With permission prompting available, should
v0.4 *also* register `bash`?

**Proposal: NO, defer.** Slack-side prompting is the foundation;
bash-via-Slack-prompt is a v0.5 decision after v0.4 has been
operationally tested. The §6.4 sandbox concerns don't go away just
because we can prompt.

**Decision:** NO, defer.

### B.4 Wire shape

- **Reuse franky's `permissions.Store`, `PermissionPrompter`, and
  `tool_permission_request` event** — no new agent API.
- **New per-thread `prompts_state.zig`**: maps `(channel, prompt_ts)
  → call_id` so reaction events on the prompt message route back to
  the agent's prompter. Mirrors the existing `reply_anchors` LRU
  pattern.
- **Extend reaction dispatch** in `bot.dispatchReaction`: when the
  reaction lands on a registered prompt message, decode the emoji →
  resolution and call `prompter.resolve(call_id, resolution)`.
- **Manifest**: no new scopes (already has `reactions:read` +
  `chat:write`).

### B.5 Tests

1. **Happy path — allow_once** — fire `tool_permission_request`,
   simulate `:white_check_mark:` reaction, assert
   `prompter.resolve(call_id, .allow_once)` was called and the agent
   continued.
2. **Always-allow promotes to store** — same with `:fast_forward:`,
   then a second prompt for the same tool runs without asking.
3. **Timeout** — spawn the prompt, sleep past the configured
   timeout, assert resolution was `.deny_once` and the explanation
   message landed.
4. **Stale reaction** — reaction lands AFTER the prompt has already
   been resolved by another path; assert no crash and a noop log
   line.

### B.6 Effort

~250-400 LOC + tests. Two days of focused work; one of which is
nailing the reaction-routing edge cases (multiple users reacting,
deletes, timing races).

## C — Session hibernation (long-term thread persistence)

### C.1 Goal

When a user comes back to a Slack thread *hours or days* later, the
bot should pick up the conversation where it left off — same model
state, same transcript, same per-thread cwd. Today:

- `bindings.json` (`(team_id, thread_ts) → ULID`) **is persisted**
  (Phase 5).
- `transcript.json` per session **is persisted** (franky's
  `coding/session.zig`).
- But the **`Agent` instance is not** — the in-memory
  `agent_cache.Cache` is unbounded and never offloads. A
  long-running bot grows without bound; on restart, a returning user
  sees "fresh context" because no one ever rehydrated the
  transcript.

The agent_cache.zig comment already flags this: *"Phase 3:
unbounded map. LRU eviction (with `agent_cache_size`) lands in Phase
5 alongside on-disk transcript reload."* Phase 5 closed the binding
+ transcript persistence but **never landed the eviction + reload**
half. v0.4 closes that gap.

### C.2 Lifecycle (proposed)

```
mention →                  cache hit? ──── yes ──> use existing Agent
                              │
                              no
                              ▼
                       bindings.json knows ULID? ── no ──> mint new ULID + new Agent
                              │
                              yes
                              ▼
                       transcript.json on disk? ── no ──> mint new Agent (binding stale)
                              │
                              yes
                              ▼
                  rehydrate Agent from transcript.json
                       (call franky.coding.session.load)
                              │
                              ▼
                       Agent ready, append new user message
```

Eviction:

- **Capacity-driven** — when `agent_cache.put` would exceed the cap,
  pick the least-recently-used Agent, persist its transcript to
  disk (it should already be persisted; this is a guard), and
  `deinit` it.
- **Idle-driven** — a background sweeper runs every N minutes;
  Agents idle for > T minutes get persisted + evicted regardless of
  cache pressure. Frees memory for bots that serve many threads.

### C.3 Open questions

#### C.3.1 — Cache cap

**Proposal:** `agent_cache_size = 16` by default (matches the
existing default in `franky-do.md` §10.2's example config).
`FRANKY_DO_AGENT_CACHE_SIZE` env var override.

**Why 16.** Each Agent holds a transcript + worker thread + small
allocator state. 16 active sessions × ~1 MiB/session ≈ 16 MiB
floor; well within VPS limits. Operators with hundreds of
concurrent threads can bump.

**Decision:** 16 with `FRANKY_DO_AGENT_CACHE_SIZE` override.

#### C.3.2 — Idle eviction threshold

**Proposal:** evict after `15 minutes` of inactivity. Long enough
that a user finishing a thought + going for coffee doesn't trigger
reload churn; short enough that a bot serving many threads frees
memory.

**FRANKY_DO_IDLE_EVICTION_MS** override. Set to `0` to disable
idle-eviction (capacity-driven only).

**Decision:** 30 minutes with `FRANKY_DO_IDLE_EVICTION_MS` override.

#### C.3.3 — Sweeper interval

**Proposal:** sweep every `60 seconds`. Cheaper than a per-session
timer; finds idle sessions within (60s + threshold).

**Decision:** 5 minutes with `FRANKY_DO_SWEEPER_INTERVAL_MS` override.

#### C.3.4 — Reload triggers

When does the bot try to reload from disk?

**Proposal: only on cache miss with binding present.** If the user
posts in a thread with no binding, treat it as a new session
(matches today's behavior). If a binding exists but no transcript
file (corrupted state), log a warn and treat as new session — don't
crash.

**Decision:** only on cache miss with binding present.

#### C.3.5 — What gets serialized

franky's `coding/session.zig` already serializes:

- `session.json` — id, model, provider, system_prompt_hash, branch
  metadata.
- `transcript.json` — message list with content blocks.

Things franky-do *also* needs:

- **bash cwd** (per-thread agent cwd) — not currently in franky's
  session schema; would need a small extension.
- **last_user_id / last_user_ms** — idle-eviction needs these.

**Proposal:** add `franky_do.json` next to session.json (same
session dir) with the franky-do-specific fields. Backward-compatible
— franky core never reads it.

**Decision:** add `franky_do.json` next to session.json (same session dir) with the franky-do-specific fields.

#### C.3.6 — Transcript size cap on reload

A 6-month-old thread might have hundreds of turns and tens of
thousands of tokens. Reloading the full transcript blows the LLM's
context window on the next call.

**Proposal:** on reload, if the transcript exceeds `0.7 ×
context_window` tokens, run franky's `coding.compaction` summarizer
synchronously before adding the user's new message. franky already
has this machinery (`/compact` slash command); reuse it. Background:
v1.5.* shipped compaction summarization.

**Open subquestion:** at what threshold? 0.7× is conservative; 0.9×
gives more raw history but risks a `context_overflow` on the very
next turn. **Sub-proposal:** 0.7 default, env-overridable.

**Decision:** 0.8× context_window on reload, env-overridable.

#### C.3.7 — Concurrent-mention safety

If two users `@`-mention the bot in the same thread within 100ms of
each other (rare but possible), and the agent is currently being
rehydrated from disk, what happens?

**Proposal:** lock the rehydrate path per-ULID. Second mention
waits for the rehydrate to complete, then runs as a normal turn.
Use `std.Io.Mutex` keyed on `(team, thread_ts)`. Mirrors print
mode's `run_mutex` pattern.

**Decision:** lock the rehydrate path per-ULID

### C.4 Wire shape

- **`agent_cache.zig`** — add LRU bookkeeping (per-key
  `last_access_ms`, doubly-linked list or just a sorted scan since
  cap is small), `evictLeastRecentlyUsed()` method,
  `evictIdleOlderThan(ms)` method, **persist on evict** hook.
- **New `agent_hibernate.zig`** — pure helper that takes an
  `Agent` + a session dir and writes/reads `franky_do.json` +
  drives `franky.coding.session.save` / `franky.coding.session.load`.
- **`bot.zig::ensureAgent`** — extend the cache-miss path: try
  `loadFromDisk(ulid)` first; only mint a fresh Agent if no
  transcript exists.
- **`main.zig`** — spawn a sweeper thread per process; tear down on
  shutdown.

### C.5 Tests

1. **Capacity eviction** — fill cache to cap+1, assert least-recent
   is evicted and persisted.
2. **Idle eviction** — set idle threshold to 100ms, set up a stale
   session, assert sweeper evicts it.
3. **Reload round-trip** — create session, persist + evict, post a
   new message, assert agent reloaded with same transcript content.
4. **Compaction on reload** — push a 100-message transcript past
   the threshold; assert `coding.compaction.summarize` ran before
   the new turn.
5. **Concurrent rehydrate** — fire two parallel mentions on the
   same `(team, thread)` mid-rehydrate; assert one rehydrate ran,
   second waited, both turns succeeded in source order.

### C.6 Effort

~400-600 LOC + tests. ~3 days. The bulk is the test suite — the
production code is mostly orchestration on top of franky's existing
session.zig + compaction.zig.

## Suggested ordering

If we're doing all three, suggested order:

1. **A — Emoji status indicators** (smallest; gives immediate
   user-visible payoff; needs the manifest scope change so
   getting it in early avoids stacked reinstalls).
2. **C — Session hibernation** (foundation for long-running
   deployments; no Slack-API changes).
3. **B — Slack permission prompting** (highest-risk for race
   conditions; benefits from the cleaner reaction routing that A
   exercises first).

Each ships its own minor version. **v0.4.0** = A; **v0.4.1** = C;
**v0.4.2** = B. Can also ship as a single v0.4.0 cut if you want
the full feature set landing together — that's a larger PR but a
smaller test matrix.

Total effort across all three: ~6-7 days of focused work.
